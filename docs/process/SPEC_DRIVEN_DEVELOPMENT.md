# 仕様駆動開発（Specification-Driven Development）

仕様を **意思決定の記録** として扱い、実装・テスト・レビューを仕様に紐づけるプロセスです。

## 1. 用語

| 用語 | 意味 |
|------|------|
| SPEC | 機能・非機能・契約（API・UI・データ）を文章化した文書。番号付きファイルを推奨。 |
| SPEC-INDEX | 全 SPEC の一覧と状態（草案・レビュー中・承認済み・廃止）。 |
| 受け入れ基準 | SPEC 内の「満たせば完了」と判定できる条件。 |

## 2. ライフサイクル

1. **草案（Draft）**: [SPEC-000-template.md](../specs/SPEC-000-template.md) を複製し、識別子（例: `SPEC-007`）と題名を付ける。INDEX に「草案」で登録。
2. **レビュー**: ステークホルダーが内容・受け入れ基準・非機能を確認。議論は PR またはレビューコメントに残す。
3. **承認（Accepted）**: 実装・テストの前提として固定。変更は改訂履歴に記載。
4. **実装**: ブランチ・PR は対応する SPEC を本文またはラベルで参照する（[GIT_WORKFLOW.md](./GIT_WORKFLOW.md)）。
5. **検証**: 受け入れ基準を満たす証跡を PR に残す（[TESTING_STRATEGY.md](../testing/TESTING_STRATEGY.md)）。
6. **廃止（Deprecated）**: 置き換え先 SPEC を INDEX に明記。旧 SPEC は履歴として残すかアーカイブ方針に従う。

## 3. 仕様に書くべきこと

- **背景と目的**: なぜこの変更が必要か。
- **スコープ**: 何を含み、何を含まないか。
- **契約**: 入出力、エラー条件、後方互換の扱い。
- **非機能**（該当時）: 性能、可用性、セキュリティ、アクセシビリティなど、プロジェクトで重要なもののみ。
- **受け入れ基準**: チェック可能な箇条書き。
- **Human-in-the-loop**: 人の判断が必須な箇所（[HUMAN_IN_THE_LOOP.md](./HUMAN_IN_THE_LOOP.md) への参照でも可）。

## 4. 仕様とコードの対応

- 1 PR が複数 SPEC にまたがる場合は、PR 本文で「各 SPEC にどう応えるか」を箇条書きする。
- 実装が SPEC から逸脱する場合は **先に SPEC を更新** するか、逸脱を明示した上でレビュー承認を得る。

## 5. AI / エージェントとの併用

- エージェントへの指示は「対象 SPEC + 変更の意図」から始める。
- エージェントは SPEC にない要件を勝手に追加しない。追加が有益でも [AGENT_OPERATING_GUIDE.md](./AGENT_OPERATING_GUIDE.md) に従いエスカレーションする。

## 6. トレーサビリティ

- リリースノートまたは CHANGELOG で、ユーザー向け変更を SPEC ID に紐づけると追跡しやすい（任意）。
