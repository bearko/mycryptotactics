# Human-in-the-loop（人の判断を組み込む）

自動化・AI エージェント・CI を進めつつ、**人が必ず介在すべき局面** を明示する。特定ツールに依存しない役割ベースの記述とする。

## 1. 目的

- 誤った自動マージ、仕様の静かな逸脱、コンプライアンス違反を防ぐ。
- 「誰が・いつ・何を承認するか」を曖昧にしない。

## 2. 典型的に人の承認が必要な局面

次のいずれかに該当する場合は、実装・マージ・本番適用を **止めて** 指名されたロールの承認を得る。

| 局面 | 理由の例 |
|------|-----------|
| 憲章・SPEC との矛盾 | 要求が文書化された合意と食い違う |
| セキュリティ境界の変更 | 認証・認可・ネットワーク境界・暗号の扱い |
| 個人情報・機微データの新規取り扱い | 法的・契約上のレビュー |
| 破壊的 API / データ互換の変更 | 利用者・移行コスト |
| 本番設定・インフラの恒久的変更 | 誤操作の影響範囲が大きい |
| 依存関係の大幅更新 | サプライチェーン・ライセンス |
| 高コスト・不可逆な操作 | 例: 大量データ削除、課金トリガー |

プロジェクトは上表を **削る・増やす** してよい。増やした項目は SPEC または運用手順にリンクする。

## 3. 承認の記録

- 承認は PR のレビュー承認、チェックリストの署名、またはチケットの状態遷移など **監査可能な形** に残す。
- 口頭のみの承認は原則認めない（緊急時は事後に文書化するルールを別途定める）。

## 4. AI / エージェントとの境界

- エージェントは上記局面を検知したら **変更を止め**、人間に要約と選択肢を返す（[AGENT_OPERATING_GUIDE.md](./AGENT_OPERATING_GUIDE.md)）。
- 人間が「続行」と言っただけでは憲章違反を正当化しない。SPEC または憲章を更新したうえで再度レビューする。

## 5. 緊急時（インシデント）

- インシデント対応で通常ルールを短縮する場合は、**事後** にインシデントレポートへ「短縮した承認」と「恒久対策」を記載する。

## 6. エスカレーション経路

- 一次: レビュアー / オンコール
- 二次: プロダクトオーナーまたは技術責任者（[PROJECT_CHARTER.md](../charters/PROJECT_CHARTER.md) のロール表に合わせる）
