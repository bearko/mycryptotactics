# デザイン憲章（DESIGN CHARTER）

UI / UX 設計で **何を譲らないか** を集約した憲章です。SPEC-002 §10 で「機能追加時に参照」と書かれた体験設計方針を、複数の SPEC を横断して守る **原則** に昇格させます。

個別の機能仕様（SPEC-NNN）は本憲章と矛盾してはいけません。矛盾する場合は本憲章を改定するか、SPEC を改定して整合を取ります。

## 1. 設計の核（What we will not break）

### 1.1 モバイルでスクロールせずに状況がわかる

戦闘中の主要情報（HP、PHY、INT、AGI、ガード、シールド、敵の意図、エナジー、手札、直近のログ）は、**iPhone SE 級（375px）の縦画面で縦スクロールせずに見える** こと。スクロールしないと読めない情報があるなら、それは「優先度が低い情報」を一画面に詰め込みすぎているサイン。

**根拠**：本作はモバイルで遊ばれる想定が強い。プレイ中にスクロールが必要だと、テンポが落ちて戦術判断の集中が切れる。

### 1.2 関連する情報は近づける（Proximity）

意味的に結びつく情報は画面上でも近くに配置する。例：

- エナジーは手札の **直上**（カードを撃つコスト原資なので）
- 敵の NEXT ACTION は敵ポートレートの **直下**（誰の予告かを一目でわかるように）
- ログは **アリーナと手札の間**（直前の操作結果がそこに出る）
- 数値ポップ（±10 など）は変化したステータス値の **上**

逆に、視覚的に派手だからといって関連のない情報を画面中央に置かない。

### 1.3 テンポを落とさないフィードバック

プレイヤーが操作した瞬間に「何が起きたか」がわかること。フィードバックは多層構造：

1. **即時**：ヒットフラッシュ・色・SE・数値ポップ（0〜0.5 秒以内）
2. **遅延**：ログのテキスト（証跡として）

即時フィードバックが弱いと「効いたのかわからない」、遅延フィードバックがないと「何が起きたか思い出せない」。両方必要。

### 1.4 数値の検証可能性

カードを撃った結果、何がどう変わったかを **画面で確認できる** こと。バフ・デバフで変動する HP / PHY / INT / AGI / ガード / シールドは、戦闘中ずっと表示しておく（隠さない）。

**根拠**：ローグライクは「不確実性を楽しむ」ジャンルであって、「自分の状態がわからない」を楽しむジャンルではない。プレイヤーは常に自分の状況を把握している前提でリスクを取る。

### 1.5 ピクセルアートをぼかさない

低解像度スプライトは **ニアレストネイバー系**（CSS `image-rendering: pixelated; image-rendering: crisp-edges;`）でのみ拡大する。バイキュービック・バイリニアは禁止。

**根拠**：素材リポジトリ（mycryptoheroes）の README にも明記されているドット絵の標準的扱い。ぼかすと作品全体の印象が壊れる。別アルゴリズムが必要なら **プリレンダ** で対応し、ブラウザでの拡大では使わない。

### 1.6 用語ラベルで「誰の情報か」を明示

敵専用の UI には `NEXT ACTION` などの英字ラベルを置く。日本語混在の画面でも、ラベルが英字なら「これは敵側の情報」と即わかる。**ラベルは省略しない**（小さくしてもよい）。

**根拠**：味方カードと敵意図の見分けがつかないと、初見の人がカードを撃つ前に固まる。

### 1.7 SPEC ファースト

UI を変えるなら、**先に SPEC を更新する**。実装が先行すると「なぜそうなっているか」が消える。SPEC-002 §10 の改訂がないままレイアウトを変える PR は受け入れない。

## 2. 視覚言語

### 2.1 カラー

| 用途 | 値 | 用途 |
|---|---|---|
| 背景 | `--bg: #121018` | ベース |
| パネル | `--panel: #1e1a28` | カード・モーダル |
| ボーダー | `--border: #3d3550` | 区切り |
| テキスト | `--text: #e6e0f0` | 標準 |
| ミュート | `--muted: #8a8199` | 補助テキスト |
| アクセント | `--accent: #c4a35a` | 注目（ヘルプ・ボス・通貨） |
| エナジー | `--energy: #5ab4c4` | エナジー・コスト |
| 危険 | `--danger: #c45c6a` | 戦闘ノード・敵 |
| 増加 | `--gain: #5ecf8a` | 数値増加・回復 |
| 減少 | `--loss: #ff6b6b` | 数値減少・被弾 |

**追加するときの原則**：新色は CSS 変数に追加してから使う。インラインで `#xxx` を書かない。

### 2.2 タイポグラフィ

- フォント：システムフォント `"Segoe UI", "Hiragino Sans", sans-serif`（Web フォントを読み込まない）
- サイズ単位：rem ベース。固定 px は最小限
- カード本文は `clamp()` で可変。狭い画面で破綻しない

**根拠**：Web フォントを使うとロードが遅くなる。Vercel フリー範囲・モバイル 3G を想定するとシステムフォントが安全。

### 2.3 スペーシング

- 戦闘画面はコンパクト：`padding: 0.5rem 〜 1rem`
- カード間隙：0.55rem
- パネル間隙：0.75rem

**根拠**：余白を増やすとモバイルで縦スクロールが必要になる。SPEC-002 §10-1「装飾用の大きなヘッダーは避ける」の延長。

### 2.4 アニメーション

- 必要なものだけに使う：ヒットフラッシュ（0.35 秒）、数値フロート（0.85 秒）、カットイン（0.45 秒）
- ターン進行中の長い演出は禁止：プレイヤーが待たされる
- `prefers-reduced-motion` の対応：将来的には対応するが、本リリースの非目標

## 3. 音

### 3.1 BGM

- 戦闘中のみループ（`pvp.mp3`）
- マップ・報酬・ショップでは無音
- 音量は控えめ（プレイヤーの環境を尊重）

### 3.2 SE

| 状況 | SE |
|---|---|
| 単体ダメージ（味方の通常攻撃ヒット、敵の通常被弾） | `1_single_damage` |
| 範囲・特殊ダメージ | `2_area_damage` |
| 回復 | `3_heal_resurrection` |
| バフ | `4_buff` |
| デバフ | `5_debuff_status_effect` |

### 3.3 ジングル

- ラン全層クリア → `clear.wav`（1 回）
- 勝利演出 → `win.mp3`
- 敗北演出 → `lose.mp3`

### 3.4 自動再生制約

iOS Safari は最初の操作前に音声を再生できない。これは仕様。**ユーザーの最初のタップ（ノード選択など）以降に音が鳴る** 挙動を許容し、エラーログを出さない。

## 4. 情報ヒエラルキー（戦闘画面）

戦闘中の縦の流れ：

```
[リソース行（GUM・HP・層・?）]   ← ラン全体の状況
[アリーナ（味方 vs 敵 + NEXT ACTION）]  ← 戦闘の主役
[ログ（直近 2〜3 行）]                ← 何が起きたか
[エナジー行]                          ← これからの選択肢の原資
[手札（3 列 × 最大 2 行）]            ← これからの選択肢
[フッター（山・捨札・ターン終了）]     ← 補助操作
```

この順序は **動かさない**。動かすと「視線が止まらない＝判断できない」状態になる。

## 5. アクセシビリティ（最低限）

リリース版で守るのは以下：

- 色だけで情報を区別しない（HP は数値・バー・色の三重）
- フォントサイズはユーザー設定を尊重する rem ベース
- ヘルプは Esc / 背景クリックで閉じられる
- タップ領域は最低 44 × 44 px（iOS HIG 準拠）

**スコープ外**：

- スクリーンリーダー対応（SPEC-005 以降）
- キーボードナビゲーション（同上）
- 文字サイズ切り替え UI（同上）

## 6. やらないこと（明示）

UI 設計で誘惑されがちだが、本作では **しない** 判断：

- 派手なパーティクルやスクリーンエフェクト（テンポを下げる）
- カットインの長い演出（操作可能になるまでが遠い）
- 通知バッジや赤丸（プレイ中に煩わしい）
- スプラッシュスクリーン（モックアップ的な印象を出さない）
- 著作権素材を流用したフォント・SE（ライセンス事故）

これらが必要だと感じたら、まず SPEC を起こして議論する。

## 7. デザイン議論の場

UI 変更は GitHub PR で行う。デザインスケッチは Figma など外部に置いてもよいが、**最終的な真実の源泉は SPEC とコード** とする。スケッチだけ更新してコードを書かない、または逆も避ける。
