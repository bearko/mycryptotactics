# 開発憲章（DEVELOPMENT CHARTER）

開発の進め方・品質・安全に関する約束事です。本作は 1 名運用なので「チーム」を想定した冗長なルールは置かず、**未来の自分・Claude Code・第三者コントリビューター** が同じ流儀で動けることを目指します。

特定ツール名（Claude Code、Cursor、Vercel）に依存しない原則を上位に置き、ツール固有の手順は別ドキュメントに切り出します。

## 1. 品質原則

### 1.1 仕様と実装の一致

リリースに含まれる挙動は、対応する SPEC に記載されているか、意図的な差分が PR / SPEC に説明されていること。**「動いてるからいい」は通さない**。

実装が SPEC からズレていることに気づいたら、まずやるのは：

1. SPEC を更新する PR を出す（実装よりこちらが先）
2. それが通ってから実装の PR を出す

逆順にしない。逆順にすると後で「なぜそうなっているか」が消える。

### 1.2 回帰の明示

既存挙動を変える PR は、影響範囲とテスト・確認の証跡を必ず残す（[TESTING_STRATEGY.md](../testing/TESTING_STRATEGY.md)）。

「自分しか触らないから大丈夫」は通さない。3 ヶ月後の自分は他人。

### 1.3 可観測性（無料・1 名運用版）

Vercel フリーで動かす制約上、APM や集約ログは置けない。最低限：

- 致命例外は `try-catch` でブラウザコンソールに出す
- ゲーム進行が止まる箇所には `console.warn` で目印をつける
- 本番ログ収集は **しない**（本作のスコープ外）

それ以上が必要になったら別 SPEC で扱う。

## 2. 安全とコンプライアンス

### 2.1 秘密情報

- 鍵・トークン・個人情報は **コミットしない**。本作はバックエンドなしのため、そもそも秘密情報を扱わない構成にする
- 万が一コミットしてしまったら、**履歴ごと書き換え**（git filter-repo や BFG）して push -force する。秘密情報は revert で消えない

### 2.2 ライセンスと素材

- 外部素材（画像・音声）の利用は [HUMAN_IN_THE_LOOP.md](../process/HUMAN_IN_THE_LOOP.md) で人が判断する
- mycryptoheroes 素材は **MCH Co.Ltd. のデザインガイドライン準拠**。リリース直前に最新ガイドラインを確認すること
- 法人・営利での利用、ガイドライン基準を超える利用は事前連絡が必要。本作は非営利・無料の二次創作として運用

### 2.3 外部依存

- npm パッケージなどのランタイム依存は **追加しない**（現状ゼロ）
- どうしても必要なら、ライセンス・メンテナンス状況・ファイルサイズを SPEC に記載する
- 開発時のみ使うツール（lint など）は別扱い。`devDependencies` に入れる

## 3. コラボレーション

### 3.1 1 名運用での非同期ファースト

「自分の頭の中にある」を許さない。判断材料は PR 本文・SPEC・チケットに書く。**3 ヶ月後の自分にも、Claude Code の新セッションにも伝わる粒度** で書く。

### 3.2 レビューの基準（self-review）

PR を作ったら、merge する前に **時間をおいて読み返す**（最低 1 時間。可能なら半日）。新鮮な目で読み直すと、自分の文章のどこが伝わっていないかわかる。

第三者にレビューしてもらえる機会があれば積極的に依頼する。レビューは行動ではなく成果物に焦点を当てる（自分にも他人にも）。

### 3.3 Human-in-the-loop

[HUMAN_IN_THE_LOOP.md](../process/HUMAN_IN_THE_LOOP.md) に定める局面では、自動化（Claude Code の自律実行）を止めて自分（人間）の判断を入れる。具体例：

- 憲章・SPEC の改訂
- ライセンス判断
- リリース当日の本番 Promote
- 第三者にプレビュー URL を見せる行為

## 4. AI / エージェントの利用

### 4.1 範囲

エージェント（Claude Code 等）は [AGENT_OPERATING_GUIDE.md](../process/AGENT_OPERATING_GUIDE.md) と [CLAUDE.md](../../CLAUDE.md) の範囲で自律してよい。それ以外は人の確認を取る。

### 4.2 生成コードの扱い

エージェントが生成したコードは **人間が書いたコードと同じ基準** でレビュー・テストする：

- SPEC との整合
- 数値・閾値の根拠
- エラーハンドリング
- 既存コードとの一貫性（命名、構造）

「Claude が書いたから正しい」は通さない。

### 4.3 SPEC を超えない

エージェントは SPEC にない要件を勝手に追加しない。便利そうな機能を思いついても、SPEC を起こす提案は人にする。提案は歓迎、勝手な実装は不可。

### 4.4 変更権限

憲章や SPEC を変更する権限は **人間のみ**。エージェントが憲章ファイルを直接編集することは認めない（Claude Code の自動実行で `docs/charters/*.md` を書き換えるのは禁止）。提案 → 人が編集、または提案 → 人がレビュー → エージェントの編集を承認、の流れにする。

## 5. 技術的負債

### 5.1 意図的なショートカット

「今は時間がない」で取ったショートカットは、**SPEC か Issue に「期限・理由・解消条件」を書いて残す**。例：

```
## 技術的負債：チャプター生成の決定論性

- 期限: SPEC-005 で対応
- 理由: 3 日リリースのため Math.random() を直接使用
- 解消条件: シード付き乱数に置き換え、テスト可能にする
```

### 5.2 無期限 TODO の禁止

`// TODO:` をコードに残さない。残すなら Issue 番号を付ける：`// TODO(#42): ...`。Issue 化されていない TODO は、PR レビューで止める。

## 6. テスト

詳細は [TESTING_STRATEGY.md](../testing/TESTING_STRATEGY.md)。原則：

- ロジック純関数（`battle-mch.js` など）は Node 標準 `assert` でユニットテストを書ける
- UI 挙動は手動 QA（PR 本文にチェックリストを貼る）
- 自動 E2E はリリース版のスコープ外

## 7. 改定

### 7.1 改定の手続き

本憲章の改定は Pull Request で行う。改定 PR では：

- 何を変えるか
- なぜ変えるか
- どの SPEC / 既存コードと整合するか
- 既存の実装に影響があるか

を本文に書く。

### 7.2 PROJECT_CHARTER との関係

本憲章は [PROJECT_CHARTER.md](PROJECT_CHARTER.md) と矛盾しない範囲で運用する。矛盾が見つかれば、まず PROJECT_CHARTER を改定する。
