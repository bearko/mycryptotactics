# 開発憲章（DEVELOPMENT CHARTER）

開発の進め方・品質・安全に関する **チーム全体の約束** です。特定ツール名に依存せず、原則として記述する。

## 1. 品質原則

- **仕様と実装の一致**: リリースに含まれる挙動は、対応する SPEC に記載されているか、意図的な差分が PR / SPEC に説明されていること。
- **回帰の明示**: 既存挙動を変える場合は、影響範囲とテスト・確認の証跡を残すこと（[TESTING_STRATEGY.md](../testing/TESTING_STRATEGY.md)）。
- **可観測性**: 障害調査に必要なログ・メトリクス・トレースの方針はプロジェクトで定める。定義がない場合は SPEC または運用ドキュメントに追記してから実装する。

## 2. 安全とコンプライアンス

- 秘密情報（鍵、トークン、個人情報）をリポジトリにコミットしない。検出された場合は直ちにローテーションと履歴対応の手順に従う。
- 外部依存の導入は、ライセンスと供給元の信頼性を確認する（プロジェクトのポリシーに従う）。

## 3. コラボレーション

- **尊重**: レビューは行動ではなく成果物に焦点を当てる。
- **非同期ファースト**: 判断材料は PR 本文・SPEC・チケットに書き、口頭だけに残さない。
- **Human-in-the-loop**: [HUMAN_IN_THE_LOOP.md](../process/HUMAN_IN_THE_LOOP.md) に定める局面では、自動化を止めて人の承認を得る。

## 4. AI / エージェントの利用

- エージェントは [AGENT_OPERATING_GUIDE.md](../process/AGENT_OPERATING_GUIDE.md) の範囲で自律してよい。
- 生成コードは人間と同じレビュー・テストの基準に置く。憲章や SPEC を変更する権限は、チームが明示したロールに限定する。

## 5. 技術的負債

- 意図的なショートカットは「期限・理由・解消条件」を SPEC またはチケットに残す。
- 無期限の TODO は避け、バックログ項目に昇格させる。

## 6. 改定

- 本憲章の改定は [PROJECT_CHARTER.md](./PROJECT_CHARTER.md) と矛盾しない範囲で行う。
