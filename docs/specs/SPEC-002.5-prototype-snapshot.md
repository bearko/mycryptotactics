## なぜこの SPEC を作るか

SPEC-002 は意図と受け入れ基準を定めていますが、**具体的な数値・配置・閾値**（手札カード幅、HP の初期値、敵意図のローテ周期など）はコード側にしかありません。3 日後の一般公開に向けて、

1. リリース時に「なぜこの値にしたか」が辿れること
2. コンテンツ追加（SPEC-004）が既存挙動を壊さないように現値を参照できること
3. ジュニア 1 名運用で、コードを毎回読み直さずに数値変更ができること

を満たすため、**現状の固定値とコメントの根拠を集約**します。コードの直接の引用ではなく、コードのコメント・コミットメッセージ・SPEC-002 の議論から **「なぜそうしたか」を残します**。

> 注：この SPEC はプロトタイプを書き換えるものではありません。**いまある挙動の言語化**です。SPEC-002 と矛盾する記述があれば、SPEC-002 が優先します。

## 1. ファイル構成（プロトタイプ）

```
prototype/
├── index.html              ← ビュー定義・CSS・エントリー
└── js/
    ├── main.js             ← 全体制御・状態遷移・SE/BGM 再生
    ├── battle-mch.js       ← MCH 式戦闘の純関数（ダメージ、回復、クリ、ガード/シールド）
    ├── cards.js            ← カードマスタ（id, name, ext, cost, target, effect）
    ├── enemies.js          ← 敵マスタ（id, hp, phy, int, agi, intent ローテ）
    ├── maps.js             ← マップグラフ生成（層・ノード種別）
    └── ui-fx.js            ← ヒットフラッシュ、stat フロート、ポートレート FX
```

> ファイル名は SPEC-002 の方針（`prototype/js/cards.js` に集約、`main.js` を薄く）から導出。リポジトリ実体に存在しないファイルがある場合は、SPEC-002 §3「目標」の「カード定義が `prototype/js/cards.js` に集約され、`main.js` が薄くなる」に向けて Day 1 でリファクタする。

## 2. 状態遷移（ビュー）

ゲームは `showView(name)` で 4 つのビューを切り替えます。

```
[map] ──選択──> [combat] ──勝利──> [reward] ──選択 or skip──> [map]
                    │                                            │
                    └──敗北──> [gameOver]                         └──ボスクリア──> [gameOver(Win)]
```

- `combat` 入場時：BGM `Audio/BGM/pvp.mp3` を `loop` で再生
- `combat` 退場時：BGM 停止
- `reward` で全層クリア表示時：SE `Audio/SE/clear.wav` を 1 回再生
- `gameOver(Win)`：勝利ジングル `Audio/SE/Jingles/win.mp3`
- `gameOver(Lose)`：敗北ジングル `Audio/SE/Jingles/lose.mp3`

**意図（SPEC-002 §10-7）**：BGM はループ、SE はワンショット。テンポを落とさずに「何が起きたか」を補強する。

## 4. ターン進行

```
ターン開始
  ├─ 味方ガード = 0 にリセット
  ├─ エナジー = 上限値（基本 3）
  ├─ 手札を 5 枚までドロー（山足りなければ捨札シャッフル）
  └─ 敵の NEXT ACTION を表示

ユーザー操作
  ├─ カードをプレイ → エナジー消費 → 効果適用 → ログ追加 → 数値フロート/FX
  └─ ターン終了ボタン

ターン終了
  ├─ 手札を捨札へ
  ├─ 敵が NEXT ACTION の通り行動 → 効果適用 → ログ追加
  ├─ 敵 HP が 0：勝利 → reward へ
  ├─ 味方 HP が 0：敗北 → gameOver へ
  └─ 続行：次ターンの NEXT ACTION を抽選 → ターン開始へ
```

**意図**：シンプルな TBS。Slay the Spire 系の慣習に乗ることでチュートリアルを最小化。

## 5. 敵意図（NEXT ACTION）

ラベルは `NEXT ACTION`、表示位置は **敵ポートレート直下**（SPEC-002 §5）。本文は以下のフォーマット：

| パターン | 表示例 | 効果 |
|---|---|---|
| 通常攻撃（物理） | `ATK PHY 100%` | 攻撃側 PHY × スキル% に対象 PHY カット率を適用したダメージ |
| 通常攻撃（魔法） | `ATK INT 120%` | 上記の INT 版 |
| 防御 | `GUARD +8` | 自身ガードを +8 |
| 特殊（HP 割合） | `SPECIAL maxHP 20%` | 対象最大 HP の 20% をダメージ。**シールドが先に吸収** |
| 回復 | `HEAL self` | `(術者INT + 対象PHY) / 2` を回復（SPEC-002 §11） |
| バフ/デバフ | `BUFF PHY +5` | 対象の PHY ステータスに +5（持続はターン制） |

**意図（SPEC-002 §10-5）**：英字ラベルで「これは敵の予告」と一目でわかるようにする。日本語表示にすると味方カードと混じる。

## 6. battle-mch.js の式（純関数）

SPEC-002 §11 と MCH 公式ヘルプを再掲＋プロト差分を整理。

### 6.1 PHY/INT 依存ダメージ

```
カット率 = min(40, floor(対象ステ / 2))   [%]
基本ダメージ = floor(攻撃側ステ × スキル% / 100 × (100 - カット率) / 100)
最終ダメージ = max(0, 基本ダメージ - 対象ガード)
```

- ガード（PHY/INT 攻撃に対する固定バリア）はダメージ計算の最後に減算
- シールドは PHY/INT ダメージには無効

### 6.2 クリティカル

```
クリ率 = min(45, floor(攻撃側 AGI / 2))   [%]
クリ時の追加分 = 戦闘開始時の基礎ステで上記式を再計算（バフデバフ無視）
合算ダメージ = 通常ダメージ + クリ追加分
```

**意図（SPEC-002 §11.1）**：原作の AGI チャージ機構はターン制と相性が悪いので廃し、AGI をクリ率に流用。PHY/INT 別々のクリ率にしないのはターン制での煩雑さ回避（必要なら別 SPEC で分離）。

### 6.3 特殊ダメージ（SPECIAL）

```
特殊ダメージ = floor(対象 maxHP × 割合% / 100)
シールド吸収 = min(対象シールド, 特殊ダメージ)
HP への到達分 = 特殊ダメージ - シールド吸収
```

**意図（SPEC-002 §11）**：シールドの存在意義を「敵の SPECIAL を防ぐ」に集約。普通の戦闘では出番がないが、ボスで効く設計。

### 6.4 回復

```
INT カードによる回復 = floor((術者 INT + 対象 PHY) / 2)
PHY カードによる回復 = floor((術者 PHY + 対象 INT) / 2)
```

公式ヘルプ通り。プロト差分なし。

## 7. UI レイアウトの固定値（SPEC-002 §10 由来）

| 要素 | 値 | 意図 |
|---|---|---|
| 手札 grid | `repeat(3, 1fr)`（3 列固定） | 5 枚を 3 列 × 2 行に収め、狭い縦画面で縦スクロール不要 |
| カード幅（`--card-w`） | 112px | 3 列 × 112px ≒ 352px。iPhone SE 級の幅でも収まる |
| カード高さ（`--card-h`） | 216px / 200px（420px 以下） | 効果テキスト 35% 領域を確保 |
| アリーナ最小高さ | 300px | ポートレート + 数値表示 + NEXT ACTION の最小 |
| ログ最大高さ | 72px | 親指エリアでスクロールせず直近 2〜3 行を読める |
| エナジー表示 | `.hand-energy-row`（手札直上） | 手札と関連情報の近接配置（§10-2） |
| ヘルプ ? ボタン | リソース行の右端 | トップバーから移設（v3.3）。リソース＝ヘッダーに集約 |

## 9. ピクセル補間

低解像度スプライト（ヒーロー、エクステンション、敵）には CSS で：

```css
image-rendering: pixelated;
image-rendering: crisp-edges;
```

を指定する。**バイキュービック補間でぼかさない**。

**意図（SPEC-002 §10-8）**：mycryptoheroes README にも書かれているとおり、ドット絵素材はニアレストネイバー法での拡大が標準。ぼやかすと作品全体の印象が損なわれる。

## 10. 既知の差分・未対応

SPEC-002 の受け入れ基準に対して、リリース前に確認すべき点：

- [ ] PHY/INT クリ率は同一式（AGI 流用）。原作のような別計算は SPEC-005 で検討
- [ ] AGI とチャージの関係（原作）はプロトでは廃止
- [ ] バフ・デバフはターン制に翻訳（持続ターン数）し、原作の数値範囲（1〜999）は採用するが本作の上限は控えめ
- [ ] HP 上限 1000（原作）は採用するが、本作のリーダー HP は 50〜100 程度に抑える（短いランで死亡判定が機能するため）
