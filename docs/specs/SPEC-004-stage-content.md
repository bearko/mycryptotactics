## 1. 概要

リリース版（v1.0）で遊べる **3 つの章** を定義します。各章はマップ生成ルール、敵プール、カードプール、ボスを持ちます。これは「ステージ設計を担当する」唯一の仕様で、リリース日までに本 SPEC を Accepted に移行することが必要です。

## 2. 背景と動機

ロードマップで決めたとおり、3 日のスコープでは「ゲームエンジンの追加開発」ではなく「コンテンツの拡充」だけが現実的です。`prototype/js/cards.js` `enemies.js` `maps.js` のデータを拡充することで「全ステージ遊べる」を満たします。

「なぜ 3 章か」の根拠：

- 1 章だけでは Slay the Spire 系のロードバイクの楽しみ（章移行で敵プールが変わる、カード幅が広がる）が出ない
- 5 章以上は 3 日では作れない（バランス調整だけで日が暮れる）
- 3 章は **企画メモ §4.2** で示唆された「戦国回廊／大航海の港／革命の街」と整合（ただし第 3 章の名称は本 SPEC で「決定の街」に変更し、ラスボス感を出す）

## 3. 目標と非目標

### 目標

- 章 1 → 章 2 → 章 3 を通しでクリアできる
- 章ごとに敵の手触りが変わる（章 1 は素直、章 2 は状態異常、章 3 はシールドを問う SPECIAL）
- 各章のボスが 1 体ずつ存在する
- 1 ラン 30〜45 分で完走できる

### 非目標

- 4 章以上の追加（SPEC-005 で）
- 新しいカード種別の追加（パワーカードなど）— SPEC-005 で
- メタ進行・アンロック要素

## 4. スコープ

### 含む

- マップ生成パラメータ（層数、層あたりのノード数、ノード種別の出現比）
- 敵マスタ（ID、HP、PHY、INT、AGI、意図ローテーション）
- カードプール（章ごとに追加されるカードのリスト、効果、コスト）
- ボス（章末固定エンカウント）

### 含まない

- ノードグラフのランダム生成アルゴリズム自体（既存実装を流用）
- カード／敵／ボスの **新スプライト**（既存素材から ID 指定で参照）

## 5. データ駆動の方針

ハードコードを避け、**1 ファイル 1 関心** で分離します。SPEC-002 §3「カード定義が `prototype/js/cards.js` に集約」を踏襲・拡張します。

```
prototype/js/
├── chapters.js   ← 章定義（id, name, theme, mapRules, enemyPool, cardPool, boss）
├── maps.js       ← マップ生成（章定義を受けてグラフを返す純関数）
├── enemies.js    ← 敵マスタ（id, stats, intentRota）
├── cards.js      ← カードマスタ（id, ext, cost, target, effect）
└── bosses.js     ← ボスマスタ（id, stats, phaseScript）
```

> 既存にこのファイル分離がない場合は Day 1 のリファクタで分離します。SPEC-002 §3「`main.js` を薄くする」と整合。

### 8.5 ラスボス：決定者

| 項目 | 値 |
|---|---|
| HP | 180 |
| PHY | 22 |
| INT | 18 |
| AGI | 10 |
| シールド | 30（戦闘開始時） |
| フェーズ | 2 段階 |

**フェーズ 1（HP > 50%）**

意図ローテ：

```
T1: ATK PHY 100%
T2: ATK INT 100%
T3: GUARD +14
T4: SPECIAL maxHP 18%
T5: BUFF PHY +5, INT +5
→ T1 へループ
```

**フェーズ 2（HP ≤ 50%）**

意図ローテ：

```
T1: ATK PHY 130%
T2: SPECIAL maxHP 25%
T3: ATK INT 130%
T4: SPECIAL maxHP 25%
→ T1 へループ
```

撃破でランクリア → 勝利演出 → ジングル `win.mp3` → タイトルへ。

**意図**：

- 序盤シールドが 30 ある → プレイヤーが「シールドって何？」を学ぶ
- フェーズ 2 で SPECIAL が 1 ターンおきに来る → 自前のシールドが効くシーンを作る
- HP 180 は短い時間で勝てない数値設定。意図的に「足りない」と感じさせ、デッキ構築の重要性を教える

## 9. ノード生成ルール（章共通）

`maps.js` で実装する純関数 `generateChapterMap(chapter)` は以下を満たす：

1. 入力：`chapter` オブジェクト（章 1〜3 の定義）
2. 出力：層構造のグラフ（既存 `prototype/js/maps.js` の形式に準拠）
3. 制約：
   - 各層に必ずプレイ可能な少なくとも 1 つのパスがある（詰みノードを生成しない）
   - 層内のノード種別は §6.2 / §7.2 / §8.2 の比率を **目安** とする（厳密でなくてよい）
   - 直前のノードと同じ種別が連続しないように軽い repulsion をかける
   - 章ボスは最終層に **1 ノードのみ** 存在し、すべてのパスが収束する

## 10. データ投入手順（Day 1 〜 Day 2）

1. `prototype/js/chapters.js` を新規作成し、章 1〜3 の定義を本 SPEC §6〜§8 に従って書く
2. `enemies.js` に章 1 の 3 種を追加 → 章 1 のマップで戦闘確認
3. 章 1 のバランス調整（係数のみ）
4. 章 2 の敵・カードを追加 → 確認
5. 章 3 の敵・カード・ボスを追加 → 確認
6. ボス戦は 1 つずつ単体テスト（マップから外して直接遷移できるデバッグフラグを置くと早い）
