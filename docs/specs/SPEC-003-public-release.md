## 1. 概要

`prototype/` を **一般公開可能な完成度** に仕上げて Vercel 本番に Promote する。本 SPEC は「コードを書き換える」ものではなく、**公開して恥ずかしくない／詰まらない／責められないライン** を受け入れ基準として明文化するものです。

## 2. 背景と動機

プロトタイプ段階の URL（`mycryptotactics.vercel.app`）は既に存在し、開発者本人や招待者が触れる状態です。これを「誰が来ても遊べる・作者を信頼できる」状態に底上げします。具体的には：

- ステージ（章）が一周通せること（→ SPEC-004）
- 作者が信頼できる（README、クレジット、ライセンス、フィードバック窓口がある）
- 致命バグで止まらないこと（→ 本 SPEC の受け入れ基準）
- 制約に対する誠実な記述があること（セーブなし、無料、二次創作）

本 SPEC は SPEC-001 / SPEC-002 を **書き換えません**。SPEC-001 が体験要件、SPEC-002 がプロトタイプ実装契約であるのに対し、本 SPEC は **公開という非機能要件** を扱います。

## 3. 目標と非目標

### 目標

- ブラウザで開いた人が **「使い方を聞かずに 1 ラン回せる」**
- アセット 404、JS 例外、状態遷移の詰まりで **画面が固まらない**
- 主要なモバイル端末（iPhone Safari、Android Chrome）と PC（Chrome/Firefox/Safari/Edge）で破綻しない
- README に **プレイ URL・操作・既知の制約・フィードバック窓口・ライセンス** が書いてある
- リリース後にロールバックできる手順が書面化されている

### 非目標

- セーブ・進捗保存（localStorage 含む。SPEC-005 以降）
- アカウント・ログイン・ランキング（同上）
- 多言語対応（日本語のみ）
- E2E 自動テスト
- Lighthouse 100 点（パフォーマンス指標は best-effort）
- 本番 URL のカスタムドメイン（`*.vercel.app` のままでよい）

## 4. スコープ

### 含む

- `README.md` の整備（プレイ URL、操作、ライセンス、クレジット、既知の制約、フィードバック窓口）
- `LICENSE` の置き場確認、または不在ならアセット由来の表記方針記載
- Vercel デプロイ手順の文書化（`docs/operations/VERCEL_DEPLOY.md`）
- Day 3 のベータテスト（友人 1〜2 名）
- リリース当日の Go/No-Go チェックリスト
- ロールバック手順

### 含まない

- 新機能（カード、敵、UI）
- スプライトの新規作成
- バックエンド導入

## 5. 契約と挙動

### 5.1 配信構成

- ホスティング：Vercel フリープラン
- ビルド：なし（静的）
- ルート `/` は `/prototype/index.html` へリダイレクト（SPEC-001 §5）
- `/mock/index.html` も同上（SPEC-002）

### 5.2 アセット参照

すべて raw.githubusercontent.com 経由で `bearko/mycryptoheroes` を参照する。プロジェクト側でアセットを再ホストしない（ライセンス方針と整合）。

### 5.3 エラーハンドリング

- 画像読込失敗：ブラウザ標準の壊れアイコン表示で継続。**画面は固まらない**
- 音声読込失敗：無音で継続。**ゲーム進行は止めない**
- iOS Safari の自動再生ブロック：ユーザー操作後に音声再生（既存挙動）。エラーログをコンソールに出さない
- ランタイム例外：可能な限り try-catch でログ化。プレイヤーには「再読込してください」という最低限の表示

### 5.4 互換性

| プラットフォーム | バージョン | 確認方法 |
|---|---|---|
| iOS Safari | iOS 16+ | 実機 |
| Android Chrome | 直近版 | 実機 or DevTools |
| PC Chrome | 直近版 | 実機 |
| PC Firefox | 直近版 | 実機 |
| PC Safari | 直近版 | 実機（Mac 所有時のみ） |
| PC Edge | 直近版 | 実機 or DevTools |

**意図**：シェアの大きい上位 5 ブラウザに絞る。Internet Explorer、古い Android WebView は対象外。

## 6. 非機能要件

### 6.1 パフォーマンス（best-effort）

- 初回読込：3G 相当でも 10 秒以内に最初の戦闘に入れる（アセットは raw URL 経由なので best-effort）
- 戦闘中：60fps 目標、30fps 以下になる挙動は要 PR 化

### 6.2 アクセシビリティ（最低限）

- ヘルプオーバーレイは Esc / 背景クリックで閉じる（既存）
- フォントサイズはユーザー設定を尊重（rem ベース、既存）
- 色だけで情報を区別しない：HP は数値・バー・赤色の三重で表現（既存）
- スクリーンリーダー対応は **対象外**（本作のスコープ外、SPEC で明記）

### 6.3 セキュリティ

- 個人情報を取得しない（→ プライバシーポリシー不要）
- localStorage 不使用 → Cookie バナー不要
- 外部 API 呼び出しなし（アセットの GET のみ）

### 6.4 法務

- 二次創作。本家 [My Crypto Heroes](https://www.mycryptoheroes.net/) の MCH Co.Ltd. デザインガイドラインに準拠（[mycryptoheroes README 末尾参照](https://github.com/bearko/mycryptoheroes#画像利用ガイドライン要約)）
- 非営利・無料公開
- README とゲーム内クレジットで本家へのリンクと「二次創作」明記

## 7. Human-in-the-loop

### 9.2 自動テスト（限定）

- `prototype/js/battle-mch.js` の純関数に Node 標準 `assert` でユニットテストを置くことを推奨（任意）
- ファイル：`tests/battle-mch.test.mjs`
- 実行：`node tests/battle-mch.test.mjs`
- カバー対象：ダメージ式、カット率、クリ加算、シールド吸収、回復係数

> 意図：jest や vitest を導入しない（SPEC-002 の「ビルドツールの導入は非目標」維持）。標準 `assert` なら追加依存ゼロで「式が壊れていないか」だけは検証できる。

## 10. デプロイ・ロールバック

### 10.1 デプロイ

`docs/operations/VERCEL_DEPLOY.md`（リリースまでに作成）に手順を書く。骨子：

1. 作業ブランチで PR を作る
2. Vercel が自動でプレビュー URL を生成する
3. プレビュー URL で smoke test
4. main へマージ
5. Vercel ダッシュボードで Production に Promote

### 10.2 ロールバック

1. Vercel ダッシュボード → Deployments → 1 つ前の Production Deployment を選択 → "Promote to Production"
2. 数十秒で本番が前バージョンに戻る
3. 後追いで `git revert` を main にマージ（履歴の整合）

### 10.3 致命バグ発覚時の判断基準

- プレイヤーが進行不能：即ロールバック
- 表示崩れ・誤字：次のリリースで修正
- 音声が出ない：次のリリースで修正
