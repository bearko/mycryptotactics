# テスト戦略（汎用）

特定フレームワークに依存しない **方針** と **証跡の残し方** を定義する。具体的なコマンド名はプロジェクトの README / CI 設定に置く。

## 1. テストの階層（概念）

| 階層 | 目的 | 典型例（スタック非依存） |
|------|------|---------------------------|
| 単体 | モジュール・関数の契約 | 小さな入力に対する期待出力 |
| 統合 | コンポーネント間の接続 | DB・メッセージキュー・外部 API のテストダブルまたはサンドボックス |
| 契約 | 公開 API の後方互換 | スキーマ差分、消費者駆動の契約テスト（任意） |
| E2E / システム | ユーザージャーニー | ブラウザまたは CLI によるシナリオ |
| 探索的 | 仕様の穴を見つける | 手動セッション（記録付き） |

プロジェクトが小さい場合は階層を省略し、**どこまで自動化するか** を一文で決める。

## 2. いつ何を書くか

- **新機能**: 受け入れ基準（SPEC）をテスト名またはチェックリストに対応させる。
- **不具合修正**: 再発防止のため **失敗するテストを先に** 追加できれば理想的（難しければ手動再現手順を PR に残す）。
- **リファクタ**: 挙動を変えない範囲は既存テストが緑であることを証跡とする。

## 3. テスト証跡（Traceability）

レビュアと将来の自分が **「何をもって正しいと言えるか」** を追えるようにする。

### Pull Request に含める（推奨）

- 実行したテストコマンドまたは CI ジョブへのリンク（緑であること）。
- 手動確認した場合: 環境、手順、期待結果、実際の結果（スクショは任意、テキストで足りることも多い）。
- SPEC の受け入れ基準チェックボックスの更新（該当すれば）。

### SPEC またはチケットに残す（任意）

- 探索的テストの所見、未カバリティ、フォローアップ課題。

## 4. カバレッジと品質指標

- カバレッジ数値は **閾値の唯一の指標** にしない。重要領域の欠落がないかレビューで確認する。
- 静的解析・型チェックを使う場合は、新規警告を増やさない方針をチームで決める。

## 5. テストデータと秘密

- 本番データをテストに使わない。必要ならマスク手順を文書化する。
- 認証情報はシークレットストアから注入し、ログに出さない。

## 6. フレーキー（不安定）なテスト

- フレーキーは優先度高の不具合として扱い、無効化は一時的かつチケット紐づけで行う。
